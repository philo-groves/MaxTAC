# Radare2 Utilities

Use radare2 utility tools for the small, high-friction tasks that appear during
binary vulnerability research: converting numbers and byte strings, assembling
or disassembling one-off instructions, hashing whole files or ranges, encoding
and decoding test data, and generating controlled buffers for lab reproduction.

This reference focuses on `rax2`, `rasm2`, `rahash2`, and the safer research
uses of `ragg2`. Prefer these tools when they make evidence more exact and
repeatable. Avoid turning utility output into an exploit artifact unless the
engagement explicitly authorizes that level of proof.

## Quick Commands

Conversions and expression math:

```text
rax2 3+0x80
rax2 -d "1<<8"
rax2 =10 0x46
rax2 =16 70
rax2 -u 389289238
rax2 -w 16 0xffff
rax2 -e 0x12345678
rax2 -n 0x1234
rax2 -N 0x1234
```

Bytes, strings, and encodings:

```text
rax2 -s 41414100
rax2 -S AAAAA
rax2 -S < input.bin > input.hex
rax2 -D SGVsbG8=
rax2 -E hello
rax2 -B hello
rax2 -b 01001000 01101001
```

Assemble and disassemble:

```text
rasm2 -L
rasm2 -a x86 -b 64 "xor eax,eax; ret"
rasm2 -a x86 -b 64 -d 31c0c3
rasm2 -a x86 -b 64 -D 31c0c3
rasm2 -a x86 -b 64 -E 31c0c3
rasm2 -a x86 -b 64 -A 31c0c3
rasm2 -a x86 -b 64 -j -D 31c0c3
rasm2 -a arm -b 16 -d 7047
```

Hash and encode:

```text
rahash2 -L
rahash2 -a sha256 sample.bin
rahash2 -a md5,sha1,sha256 sample.bin
rahash2 -a sha256 -s "hello world"
rahash2 -B -b 1M -a sha256 dump.bin
rahash2 -E base64 -s hello
rahash2 -D base64 -s SGVsbG8=
rabin2 -K sha256 -S sample.bin
```

Controlled test buffers and snippets:

```text
ragg2 -h
ragg2 -L
ragg2 -P 256
ragg2 -q <offset>
ragg2 -B 41414141 -S marker -f raw
ragg2 -a x86 -b 64 -e "main@global(){ return(0); }"
ragg2 -a x86 -b 64 -f c test.r
```

Useful in-session equivalents:

```text
? 3+0x80
?v 0x1000+0x40
pa xor eax,eax
pad 31c0c3
pae 31c0c3
ph sha256 0x100 @ 0x401000
p8 32 @ 0x401000
pc 32 @ 0x401000
```

## Tool Selection

Use `rax2` when the question is representation:

- Convert decimal, hex, octal, binary, ternary, units, timestamps, floats, and
  signed values.
- Swap endian representations.
- Convert raw bytes to hexpairs and hexpairs to raw bytes.
- Encode or decode base64.
- Produce r2-style numeric output for scripts.

Use `rasm2` when the question is instruction encoding:

- Assemble a small instruction sequence.
- Disassemble hexpairs or raw bytes outside a full r2 session.
- Verify architecture, bitness, endianness, syntax, or CPU variant.
- Convert instruction bytes to ESIL or analysis metadata.
- Produce JSON, C arrays, or r2 command output for scripts.

Use `rahash2` when the question is identity or transformation:

- Hash a file, string, stream, range, or fixed-size blocks.
- Compare firmware, RAM dumps, memory ranges, or sections.
- Encode and decode data such as base64.
- Test cryptographic primitives in a controlled way.
- Produce reproducible checksums for evidence.

Use `ragg2` when the question is controlled test material:

- Generate cyclic patterns and locate overwrite offsets.
- Build small, non-weaponized code snippets for lab-only harnesses.
- Append fixed bytes, strings, numbers, padding, or file contents.
- Render generated bytes as raw, C, Python, JavaScript, PE, ELF, or Mach-O.
- Patch generated buffers at known offsets.

When in doubt, prefer the smallest tool that answers the question. For example,
use `rax2 -e` for endian conversion, not a custom script; use `rasm2 -D` for a
3-byte opcode, not a full analysis project.

## Rax2 Calculator and Converter

`rax2` is a shell-friendly expression evaluator and converter. Inside r2, much
of the same expression functionality is available through the `?` command.

### Numeric Expressions

Convert between bases:

```text
rax2 10
rax2 0xa
rax2 =10 0x46
rax2 =16 70
rax2 =2 0x46
rax2 1100011b
rax2 Bx63
```

Evaluate expressions:

```text
rax2 3+0x80
rax2 -d "1<<8"
rax2 -d "0x1000 + (4 * 7)"
rax2 -k 33+3
```

Interpret units:

```text
rax2 2M
rax2 -u 389289238
```

Use these when translating:

- File offsets to virtual-address deltas.
- Field sizes from decimal specs into hex.
- Alignment and mask expressions.
- Length checks and allocation sizes.
- Crash offsets and cyclic-pattern offsets.

### Signed, Word, Float, and Timestamp Values

Signed and word-sized values:

```text
rax2 -77
rax2 0xffffffb3
rax2 -w 16 0xffff
rax2 -w 32 0xffffffff
```

Float conversion:

```text
rax2 3.33f
rax2 Fx40551ed8
rax2 -f 6.3+2.1
```

Timestamps:

```text
rax2 -t 1234567890
```

Use these for:

- Signedness bug triage.
- Integer truncation and sign extension checks.
- Floating-point parser oddities.
- File-format timestamps and build metadata.

### Endian and Binary Representations

Swap endianness:

```text
rax2 -e 33
rax2 -e 0x21000000
rax2 -e 0x12345678
```

Emit little-endian binary numbers:

```text
rax2 -n 0x1234
rax2 -N 0x1234
```

Use endian conversion for:

- File-format fields.
- Network versus host order.
- Crash values copied from a debugger.
- Immediate values in instruction bytes.
- Writing exact patch or test bytes.

### Raw, Hex, ASCII, and Base64

Hexpairs to raw:

```text
rax2 -s 41414100
rax2 -s "43 4a 50"
```

Raw to hexpairs:

```text
rax2 -S AAAAA
rax2 -S < input.bin > input.hex
```

String to binary bits and back:

```text
rax2 -B hello
rax2 -b 01001000 01101001
```

Base64:

```text
rax2 -E hello
rax2 -D SGVsbG8=
```

C byte array:

```text
rax2 -i < bytes.bin
```

Use these for:

- Copying bytes between logs, debugger output, and test inputs.
- Turning crash bytes into a raw reproducer fragment.
- Decoding base64 config blobs.
- Converting suspicious embedded strings into hex search terms.
- Producing compact evidence snippets.

### Rax2 Evidence Tips

- Quote expressions containing shell metacharacters such as `<<`, `>>`, `&`, and
  parentheses.
- Record whether a value is signed, unsigned, little-endian, or big-endian.
- For binary evidence, keep both raw bytes and a printable hex representation.
- Prefer `-d` when a decimal value is clearer for lengths and offsets.
- Prefer `=16` or default hex when matching disassembly and debugger output.

## Rasm2 Assembly and Disassembly

`rasm2` is the standalone assembler and disassembler. It supports many
architectures through plugins and works well for quick opcode verification,
patch-size checks, ESIL checks, and scriptable disassembly.

### Choose Architecture and Bits

List supported plugins:

```text
rasm2 -L
rasm2 -LL
```

Set architecture and bits:

```text
rasm2 -a x86 -b 64 "nop"
rasm2 -a arm -b 16 -d 7047
rasm2 -a mips -b 32 -e -d 00000000
```

Select CPU, OS, syntax, and endian when needed:

```text
rasm2 -a x86 -b 64 -s intel -d 31c0
rasm2 -a x86 -b 64 -s att -d 31c0
rasm2 -a arm -b 32 -c cortex -d 00000000
rasm2 -a x86 -b 64 -k linux "syscall"
rasm2 -a mips -b 32 -e -d 00000000
```

Environment variables can set defaults:

```text
RASM2_ARCH=x86
RASM2_BITS=64
```

Always record `-a`, `-b`, endian, syntax, and CPU variant when opcode bytes are
part of evidence. Wrong mode is one of the easiest ways to create a false lead.

### Assemble

Assemble one instruction:

```text
rasm2 -a x86 -b 64 "xor eax,eax"
```

Assemble multiple instructions:

```text
rasm2 -a x86 -b 64 "xor eax,eax; ret"
```

Read from stdin:

```text
echo "push rax;nop;nop" | rasm2 -a x86 -b 64 -f -
```

Read from a file:

```text
rasm2 -a x86 -b 64 -f patch.asm
```

Output C format or r2 commands:

```text
rasm2 -a x86 -b 64 -C "xor eax,eax; ret"
rasm2 -a x86 -b 64 -r "xor eax,eax; ret"
```

Use assembly output for:

- Checking exact patch length.
- Replacing a branch with a same-size sequence in a local copy.
- Validating a NOP sled size.
- Building a tiny harness stub for controlled testing.
- Documenting a vendor patch in instruction bytes.

Do not write assembled bytes into a target until you have verified length,
alignment, architecture, and permissions. Use a copy for patch experiments.

### Disassemble

Disassemble hexpairs:

```text
rasm2 -a x86 -b 64 -d 31c0c3
```

Show offsets and bytes:

```text
rasm2 -a x86 -b 64 -D 31c0c3
```

Read binary input:

```text
rasm2 -a x86 -b 64 -B -l 32 -d -f bytes.bin
```

Skip bytes:

```text
rasm2 -a x86 -b 64 -i 2 -D 6690c3
```

Output JSON:

```text
rasm2 -a x86 -b 64 -j -D 31c0c3
```

Use disassembly output for:

- Verifying crash bytes from `px` or a debugger.
- Understanding short patch hunks from `radiff2`.
- Checking shellcode-like blobs in a controlled lab.
- Confirming whether data is actually code.
- Comparing architecture modes, especially ARM versus Thumb.

### ESIL and Analysis Information

Display ESIL for instruction bytes:

```text
rasm2 -a x86 -b 64 -E 31c0c3
```

Display analysis information:

```text
rasm2 -a x86 -b 64 -A 31c0c3
```

Use these as a quick pre-check before loading a full r2 ESIL session:

- Does this instruction write a register or memory?
- Does it branch, call, return, trap, or change flags?
- Is a patch changing semantics or only syntax/noise?
- Does an instruction have ESIL support for this architecture?

### Syntax and Filters

Convert or filter syntax:

```text
rasm2 -a x86 -b 64 -F att2intel -d <hexpairs>
rasm2 -a x86 -b 64 -F x86.pseudo -d <hexpairs>
```

Use filters to normalize evidence, but keep raw bytes as the source of truth.
Pseudo syntax can be easier to read but can hide important operand details.

## Rahash2 Hashing, Encoding, and Checksums

`rahash2` computes hashes and checksums for files, strings, stdin streams, and
blocks. It also exposes encoding and some crypto plugins. Use it for evidence
identity, forensics-style block comparison, section checksums, and controlled
encoding/decoding tasks.

### List Algorithms

List hash, encoding, and crypto plugins:

```text
rahash2 -L
rahash2 -L | Select-String "^h"
rahash2 -L | Select-String "^e"
rahash2 -L | Select-String "^c"
```

On non-Windows shells:

```text
rahash2 -L | grep "^h"
rahash2 -L | grep "^e"
rahash2 -L | grep "^c"
```

The available algorithms depend on the installed radare2 build. Check live
output before depending on a specific hash, encoder, or cipher.

### Hash Files, Strings, and Streams

Hash a file:

```text
rahash2 -a sha256 sample.bin
rahash2 -a md5,sha1,sha256 sample.bin
```

Hash a string:

```text
rahash2 -a sha256 -s "hello world"
```

Hash stdin:

```text
Get-Content sample.bin -Encoding Byte -Raw | rahash2 -a sha256
```

On non-Windows shells:

```text
cat sample.bin | rahash2 -a sha256
```

Use hashes for:

- Evidence file identity.
- Patch-diff pair identity.
- Reproducer input identity.
- Dump, firmware, and extracted component identity.
- Confirming that a patched test copy matches expected output.

For security evidence, prefer SHA-256 or stronger. Keep MD5/SHA1 only when
matching vendor data, legacy metadata, or existing sample databases.

### Block Hashing

Hash fixed-size blocks:

```text
rahash2 -B -b 1M -a sha256 dump.bin
```

Use block hashing for:

- RAM dumps.
- Firmware images.
- Disk or flash captures.
- Large logs or traces.
- Before/after memory snapshots.

Block hashes help identify repeated, zeroed, or changed regions. Combine them
with `radiff2`, section metadata, and offsets before making behavioral claims.

### Section Hashing with Rabin2

Hash sections through `rabin2`:

```text
rabin2 -K md5 -S sample.bin
rabin2 -K sha256 -S sample.bin
```

Use section hashes to:

- Compare firmware components.
- Detect patched code or data sections.
- Confirm that only metadata changed.
- Track unpacked or modified sections after extraction.

Section hashing complements `radiff2`; it does not replace analysis of changed
instructions or data flows.

### Encoding, Decoding, and Crypto Plugins

Encode base64:

```text
rahash2 -E base64 -s hello
```

Decode base64:

```text
rahash2 -D base64 -s SGVsbG8=
```

Use an XOR transform in a controlled lab:

```text
rahash2 -E xor -S s:password -s hello
```

Use these for:

- Decoding embedded config values.
- Testing whether a suspicious blob is base64/base91/punycode.
- Checking trivial XOR or legacy obfuscation hypotheses.
- Producing exact decoded bytes for follow-up search.

Do not treat successful decoding as proof of cryptographic weakness. Confirm
where the data is used and whether keys, IVs, modes, and integrity checks are
actually security relevant.

## Ragg2 Controlled Buffers and Snippets

`ragg2` is radare2's egg compiler and buffer generator. It can compile small
snippets, append bytes and strings, add padding, generate patterns, use encoders,
and emit output in multiple formats.

For MaxTAC usage, prioritize controlled test material:

- Cyclic/de Bruijn patterns for overwrite offset discovery.
- Padding and marker bytes for parser reproduction.
- Minimal, non-weaponized code snippets for local harnesses.
- Raw, C, Python, JavaScript, ELF, PE, or Mach-O output for tests.

Avoid using `ragg2` to generate payloads beyond the proof level authorized by
the engagement. A crash proof usually needs controlled bytes and state evidence,
not a deployable payload.

### Help and Plugins

Show help and list plugins:

```text
ragg2 -h
ragg2 -L
```

Select architecture, bits, and OS:

```text
ragg2 -a x86 -b 64 -k linux test.r
ragg2 -a arm -b 32 -k linux test.r
```

Output formats:

```text
ragg2 -f raw test.r
ragg2 -f c test.r
ragg2 -f python test.r
ragg2 -f javascript test.r
ragg2 -f elf -o test.elf test.r
ragg2 -F -o native.bin test.r
```

### Patterns and Offsets

Generate a cyclic pattern:

```text
ragg2 -P 256
```

Find an offset in the pattern:

```text
ragg2 -q <value-or-fragment>
```

Use this for stack or heap overwrite triage:

1. Generate a pattern larger than the suspected buffer.
2. Feed it through the normal input path.
3. Capture the overwritten register or memory bytes.
4. Use `ragg2 -q` to locate the offset.
5. Confirm with a second input using a marker at that offset.

Record endian and register width when converting an overwritten value into the
fragment passed to `-q`.

### Padding and Appended Data

Append bytes, files, numbers, and strings:

```text
ragg2 -B 41414141 -f raw
ragg2 -C extra.bin -f raw
ragg2 -n 0x41424344 -f raw
ragg2 -N 0x4142434445464748 -f raw
ragg2 -S marker -f raw
```

Patch generated data:

```text
ragg2 -B 4141414142424242 -w 4:43434343 -f raw
ragg2 -B 0000000000000000 -d 0:0x41424344 -f raw
ragg2 -B 0000000000000000 -D 0:0x4142434445464748 -f raw
```

Add padding:

```text
ragg2 -p n10 -f raw test.r
ragg2 -p N10 -f raw test.r
ragg2 -p a32 -f raw test.r
ragg2 -p A32 -f raw test.r
```

Padding modes from help distinguish beginning versus end:

- Lowercase padding markers such as `n`, `t`, `a`, `s` apply at the beginning.
- Uppercase padding markers such as `N`, `T`, `A`, `S` apply at the end.

Use these for reproducible input fragments, not for hiding behavior.

### Minimal Snippets

Compile a tiny ragg2 expression:

```text
ragg2 -a x86 -b 64 -e "main@global(){ return(0); }"
```

Compile from a file:

```text
ragg2 -a x86 -b 64 test.r
```

Render as C:

```text
ragg2 -a x86 -b 64 -f c test.r
```

Use minimal snippets for:

- Testing instruction encodings in an executable container.
- Building tiny syscall or ABI probes in a local lab.
- Confirming stack alignment or calling convention assumptions.
- Creating harness stubs that do not include target-specific payload logic.

Avoid built-in shellcode plugins unless the engagement explicitly calls for that
level of exploit proof. Prefer debugger and crash-state evidence for ordinary
vulnerability research.

## In-Session Equivalents

Many utilities have equivalents inside r2.

Expression math:

```text
? 3+4
?v 0x1000+0x40
?vi 0xff
```

Assemble and disassemble:

```text
pa xor eax,eax
pad 31c0c3
pae 31c0c3
pd 10
pD 32
pi 10
pI 32
pda
```

Write assembled or raw bytes only in writable sessions:

```text
wa xor eax,eax
wx 31c0c3
```

Hash current block or range:

```text
ph?
ph sha256 0x100 @ 0x401000
```

Print bytes and C/Python-friendly forms:

```text
p8 32 @ 0x401000
px 64 @ 0x401000
pc 64 @ 0x401000
pr 64 @ 0x401000
ps @ 0x402000
```

Use in-session commands when context matters:

- The address is already mapped and rebased.
- The bytes come from debugger memory.
- The function has analysis flags.
- The current block size is meaningful.
- The output should be chained into r2 commands.

Use standalone utilities when the task is independent, easier to script, or part
of evidence collection outside a project.

## Vulnerability Research Patterns

### Integer and Endian Triage

Goal: verify whether a length, offset, count, or field conversion matches the
suspected bug.

Commands:

```text
rax2 -d "0xffff + 1"
rax2 -w 16 0xffff
rax2 -e 0x12345678
rax2 -n 0x1234
rax2 -N 0x1234
```

Use for:

- Signedness conversions.
- Integer wrap before allocation.
- File-format length fields.
- Network versus host byte order.
- Immediate operands in patch diffs.

Evidence:

- Source value and base.
- Destination width and signedness.
- Endian interpretation.
- Where the value appears in bytes, registers, and code.

### Patch Byte Verification

Goal: explain a small vendor patch or local test patch.

Commands:

```text
radiff2 old.bin new.bin
rasm2 -a x86 -b 64 -D <changed-bytes-old>
rasm2 -a x86 -b 64 -D <changed-bytes-new>
rasm2 -a x86 -b 64 -E <changed-bytes-new>
```

Use `rax2` for immediates:

```text
rax2 -d 0x40
rax2 -e 0x40000000
```

Confirm in r2:

```text
r2 -A new.bin
s <changed-address>
pd 16
aoj
```

Look for branch flips, changed constants, added clamps, different call targets,
NOPs, or length calculations.

### Crash Offset Discovery

Goal: locate input offset controlling a register, return address, object field,
or faulting pointer.

Commands:

```text
ragg2 -P 512 > pattern.txt
```

After the crash, convert register bytes if needed:

```text
rax2 -e 0x41424344
ragg2 -q <fragment>
```

Confirm with a second input:

```text
ragg2 -B <prefix> -S MARK -B <suffix> -f raw > confirm.bin
```

Evidence:

- Pattern length.
- Crash register or memory bytes.
- Endian conversion if applied.
- Offset returned by `ragg2 -q`.
- Confirmation input and resulting state.

### Firmware and Memory Identity

Goal: show that two artifacts, sections, or ranges are identical or different.

Commands:

```text
rahash2 -a sha256 firmware.bin
rahash2 -B -b 1M -a sha256 firmware.bin
rabin2 -K sha256 -S component.bin
```

For changed blocks:

```text
radiff2 old.bin new.bin
rahash2 -B -b 64K -a sha256 old.bin
rahash2 -B -b 64K -a sha256 new.bin
```

Use when comparing firmware revisions, memory dumps, unpacked code, or extracted
components.

### Encoded Config or Blob Triage

Goal: test whether suspicious bytes are an encoding or trivial obfuscation.

Commands:

```text
rax2 -D <base64>
rahash2 -D base64 -s <base64>
rax2 -S < decoded.bin
rahash2 -E xor -S s:<key> -s <candidate>
```

Then search decoded bytes:

```text
rafind2 -X <hexpairs> sample.bin
r2 -A sample.bin
/x <hexpairs>
```

Evidence:

- Original bytes or string.
- Decoder or transform tried.
- Output bytes and printable strings.
- Xrefs or runtime code that uses the decoded value.

### Instruction Semantics Probe

Goal: quickly answer what an instruction sequence does before deeper analysis.

Commands:

```text
rasm2 -a x86 -b 64 -D <hexpairs>
rasm2 -a x86 -b 64 -E <hexpairs>
rasm2 -a x86 -b 64 -A <hexpairs>
```

Then use ESIL or debugging if state matters:

```text
r2 -A sample.bin
s <address>
aoj
aei
aeim
aes
```

Use this for gadget triage, patch diffs, suspicious code caves, and short decode
loops.

### Reproducer Fragment Generation

Goal: build exact bytes for a parser, file-format field, or network message.

Commands:

```text
rax2 -n 0x20
rax2 -s 41414141
ragg2 -B <header-hex> -n 0x20 -S marker -B <tail-hex> -f raw > repro.bin
rahash2 -a sha256 repro.bin
```

Keep the fragment simple and explain each field. Use `rahash2` to identify the
generated reproducer in notes and reports.

## Evidence Checklist

Collect:

- Tool name and version when available.
- Exact command line, including architecture, bits, endian, syntax, CPU, OS, and
  output format options.
- Input bytes or file hash.
- Output bytes, decoded text, hash, or disassembly.
- Interpretation: signedness, endian, width, offset, or ABI assumption.
- For `rasm2`: raw bytes plus architecture and mode.
- For `rahash2`: algorithm, block size, range, and file identity.
- For `ragg2`: pattern length, offset result, generated bytes, and proof that the
  artifact was used only in authorized lab testing.
- Any in-session context: seek address, block size, map base, and whether the
  session was writable or debugging live memory.

Prefer reproducible text artifacts:

```text
rahash2 -a sha256 sample.bin > sample.sha256.txt
rasm2 -a x86 -b 64 -D <hexpairs> > changed.disasm.txt
ragg2 -P 512 > pattern.txt
```

When output bytes are report evidence, include both a hash and a short hexdump.

## Cautions

- Always specify architecture and bits for `rasm2`; defaults can be wrong.
- ARM versus Thumb mode changes instruction meaning.
- Endianness affects both disassembly and numeric interpretation.
- Shells may interpret `<<`, `>>`, `&`, parentheses, and backticks; quote
  expressions.
- `rax2` conversions are representation aids, not proof of code behavior.
- `rahash2` weak hashes such as MD5 and SHA1 are useful for legacy matching, not
  collision-resistant evidence.
- Encoding and XOR transforms are not automatically cryptographic findings.
- `ragg2` can generate executable artifacts; keep usage scoped to authorized lab
  reproduction and avoid unnecessary payload generation.
- In-session `wa` and `wx` mutate files or memory when opened writable.
- Visual assembler changes may remain in memory until committed; record whether
  bytes were actually written.
- Generated patterns and buffers can crash targets. Use controlled environments.
